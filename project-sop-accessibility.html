🧩 Background

problem framing

In Good Manufacturing Practice (GMP) environments, all Standard Operating Procedures (SOPs) are tightly controlled. SOPs are the formal documents that tell analysts and operators exactly how regulated work should be performed, recorded, and reviewed.

In practice, these documents are essential to quality and compliance — but they are not always easy to use. Digital versions are often stored as scanned PDFs, which makes them difficult to search, cross-reference, or navigate efficiently. As a result, users may rely on file titles, memory, and manual scrolling just to find the right procedure.

💬 Usability is often invisible in compliance-heavy systems, yet it strongly shapes efficiency, confidence, and error risk in everyday work.

⚠️ Problem

document friction

Observed pain points

  • Scanned SOP files blocked keyword search.
  • Document titles were often too broad or too similar.
  • Users opened multiple wrong files before locating the correct one.
  • Urgent review situations amplified navigation burden and cognitive fatigue.

Why this was difficult

In a validated GMP environment, the source files themselves cannot be casually rewritten or transformed into a new interface. That meant the design challenge was not "replace the system," but rather "improve usability without touching the validated archive."

🎯 Design question

HCI framing

How can a human-centered retrieval layer improve navigation and reduce documentation fatigue in a regulated environment where only validated materials can be used operationally?

🧠 Method

low-code + mixed methods

1. Excel-based keyword mapping

Reviewed approximately 500 SOPs and extracted representative keywords, document identifiers, page references, and related process categories to build a searchable metadata index.

2. Structure and overlap analysis

Used Excel, Python, and R to inspect overlapping terms, recurring categories, and retrieval-relevant keyword patterns across documents.

3. Prototype artifact

Created an external retrieval aid rather than a replacement system: a prototype accessibility artifact that allowed users to search by keywords, related process, and document cues without altering the validated SOP files themselves.

4. Prototype-based usability testing

Conducted usability testing with QC users using the prototype artifact. The artifact was not deployed as a live GMP production tool; instead, it was used as a testable representation of a compliance-safe accessibility layer.

🧪 Usability testing

prototype evaluation

Testing setup

Because validated GMP environments cannot freely adopt unvalidated interface changes, the prototype was evaluated outside the live controlled system. Users were asked to complete retrieval tasks using the prototype artifact as a simulated accessibility layer.

prototype artifact task-based test QC users

What was measured

  • time to locate a target SOP
  • number of wrong-document openings
  • user confidence in identifying the right file
  • qualitative feedback on navigational clarity

Why this matters

The testing did not claim production deployment inside a validated GMP system. Instead, it asked a more HCI-relevant question: if a compliance-safe accessibility layer were available, would actual users perform retrieval work more efficiently and with less uncertainty?

A representative reaction from testing was essentially: "Now I know where to look first." That shift mattered because the intervention did not simplify the SOP content itself; it improved how users approached the archive.

📄 Paper in progress

research output

This project is being developed into a paper on accessibility, document usability, and compliance-constrained design in regulated environments. The paper extends the case study from a practical prototype into a broader HCI argument: usability improvements in validation-sensitive systems can often be achieved through external support layers rather than direct modification or replacement of the underlying system.

The paper-in-progress uses this project to frame a broader question about how human-centered design can operate in environments where formal systems are difficult to modify, yet users still experience substantial navigational and cognitive burden.

💬 Reflection

why it is HCI

This project showed that usability problems in regulated environments are not trivial inconveniences; they shape time, confidence, navigation, and cognitive load in everyday work. In other words, document control may preserve compliance, but it can also generate hidden interaction costs.

From an HCI perspective, the important insight was that meaningful design improvement did not require rewriting the validated system. An external, low-code accessibility layer was enough to improve user performance and reveal a concrete design opportunity.

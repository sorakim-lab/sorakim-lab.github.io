🧩 Background

document overload

In Good Manufacturing Practice (GMP) environments, every Deviation, CAPA, and Change Control record must be cross-linked to demonstrate process integrity and inspection readiness. These records are often deeply interdependent, but they usually exist as separate validated Word files.

Because these files cannot be freely restructured or automatically linked, analysts and QA reviewers must repeatedly copy and verify identifiers such as deviation IDs, batch numbers, product codes, status labels, and closure dates across multiple records.

What looks like a traceability requirement on paper becomes, in practice, a cognitive burden placed on people — a form of friction that accumulates invisibly across hundreds of linked records in a real GMP environment.

💬 In validation-constrained systems, people often become the missing infrastructure between documents that cannot easily talk to each other.

⚠️ Observation

field-level inefficiencies

Observed pain points

  • Users relied on memory or keyword search to link related deviations and CAPAs.
  • Inconsistent file naming increased the risk of outdated or incorrect version use.
  • QA reviewers spent excessive time checking document status and closure consistency.
  • Manual linkage work created fatigue during audits and inspection preparation.

Why this mattered

These were not individual user mistakes. They were consequences of system design under validation constraints, where automation was limited and humans became the connectors of information across fragmented records.

Each cross-reference check that should be automatic instead becomes a cognitive task — one that accumulates across hundreds of linked records in a real GMP environment.

🎯 Design question

HCI framing

How can document traceability be enhanced and cognitive load reduced without breaching validation requirements or introducing new software systems?

🧠 Method

metadata mapping + computational analysis

1. Synthetic dataset construction

Generated a synthetic GMP documentation network of 150 records (60 Deviations, 40 CAPAs, 30 Change Controls, 20 SOPs) using Python, replicating structural properties of real regulated documentation — including partial linkage, version drift, and lifecycle inconsistency.

2. Cross-reference metadata mapping

Extracted key identifier fields — Deviation ID, CAPA ID, batch number, product code, status, closure date, SOP version — and constructed a structured linkage matrix with status-based filters and category-level keyword indexing.

3. Computational linkage analysis

Used Python (pandas, networkx) to automate cross-link verification, detect missing connections, identify outdated SOP references, and surface lifecycle mismatches — producing quantified evidence of structural traceability fragmentation.

4. Network graph analysis

Modeled the documentation network as a directed graph where nodes represent records and edges represent cross-reference relationships — visualizing hub concentration, isolated clusters, and cognitive load distribution.

5. Interactive prototype + workplace evaluation

Built a Traceability Explorer in Streamlit and evaluated it with 17 GMP QC practitioners using real workplace data and structured traceability tasks. Evaluation focused on task completion time, document discovery, path completion, and navigation efficiency.

What this graph shows

Each node is a GMP record. Each directed edge is a cross-reference link — the kind of connection a reviewer must manually trace across separate validated files. The graph reveals density asymmetry (hub records concentrating cognitive work), isolated clusters (traceability gaps invisible until audit pressure), self-loops (circular linkage anomalies), and the cumulative cognitive load embedded in 3.71 average linkages per record.

Cross-reference link coverage

Deviation and SOP records have the highest cross-reference connections (130 each), while Change Control coverage is notably lower (47) — meaning downstream process consequences are often the least traceable part of the chain.

Record status heatmap

Deviations show high concentrations of both Closed and Under Review statuses simultaneously — a lifecycle pattern creating ambiguity during audit reconstruction that reviewers must resolve manually.

Why synthetic data — and why that is appropriate here

Real GMP records contain proprietary identifiers and investigation details that cannot be disclosed. The synthetic dataset was constructed to preserve the structural properties of real documentation networks without exposing organizational data. The research argument depends on network-level patterns, not specific record content.

🧪 Workplace usability study

N = 17 · GMP QC practitioners · real data

Because the traceability problem is deeply embedded in real GMP work, the evaluation was conducted with 17 QC practitioners using actual workplace documentation rather than the synthetic demo corpus. The same prototype code was applied to real records, and participants completed structured traceability tasks across three conditions.

Task design

  • Task 1 — Information retrieval: "Find the SOP relevant to this situation."
  • Task 2 — Document linkage: "Find the documents related to this SOP."
  • Task 3 — Path tracing: "Describe how you moved from the initial document to a final judgment."

What was measured

  • Task completion time (seconds)
  • Number of documents discovered
  • Path completion rate — meaningful traceability chain established
  • Click count / navigation efficiency
  • Ease of use and real-world adoption intent
real workplace data NDA — logs not disclosed

🔑 Key finding: all four measures improved substantially

Across all three tasks, the cross-reference system produced consistent improvements in speed, discovery, path completion, and navigation efficiency — while 17/17 participants rated the tool as easy to use and said they would use it in real work.

💻 Computational contribution

python · networkx · pandas

What the analysis adds

  • Automated cross-link verification across all 150 records
  • Missing-link detection for CAPA and Change Control connections
  • SOP version drift identification across referenced records
  • Lifecycle mismatch detection — closed deviations with open CAPAs
  • Network graph construction showing record-level connectivity and hub concentration

Why it matters for the HCI argument

Without computational analysis, the traceability problem can only be described anecdotally. The network graph makes cognitive burden visible — each edge is a manual lookup, each disconnected node is a gap that only a person can fill. The analysis transforms a qualitative observation into a structural, measurable claim.

The core argument

The gain here is not just efficiency. It is a redistribution of cognitive work: users no longer have to serve as the only linking mechanism across fragmented documentation. In the network graph, every edge is a task that currently sits with a person. A visibility layer does not eliminate that work — but it makes it legible, surfaceable, and therefore manageable.

💡 Future AI extension direction

visibility support, not automation

The current prototype uses rule-based linkage detection and metadata mapping. A natural next step would be to layer a lightweight AI support component — not to automate compliance decisions, but to make the reasoning behind traceability gaps more legible.

Gap explanation

An LLM layer could explain why a linkage gap is risky in plain language, given the record's context, status, and position in the documentation chain.

Related record surfacing

Surface records that share root cause categories or batch numbers but are not formally linked — expanding the visible documentation network beyond explicit cross-references.

Audit narrative generation

Generate a plain-language audit narrative from the chain, reducing the time reviewers spend reconstructing history from scratch across disconnected files.

Design principle: AI as a reasoning support layer that expands visibility into documentation — not a replacement for human judgment or validated documentation systems.

📄 Paper in progress

research output

This project is being developed into a paper on traceability, metadata infrastructure, and cognitive load in compliance-constrained documentation systems. The paper extends the cross-reference intervention into a broader HCI argument about how validation constraints shift linking work onto users — and how lightweight external support layers can make that work more visible without breaching system integrity.

In paper form, the project contributes to research on regulated knowledge systems by demonstrating that traceability is not only a technical requirement but also an interaction and cognition problem shaped by documentation architecture.

💬 Reflection

why it is HCI

This project demonstrated that compliance and usability are not mutually exclusive. Even within the constraints of validated documentation, human-centered design can restore clarity, efficiency, and cognitive equity.

The network graph is the most honest representation of what daily traceability work looks like in a GMP environment. Each edge is a manual check. Each disconnected node is a gap that a person must hold in memory. What this project asks — and begins to answer — is whether the cognitive burden of that work has to remain invisible.
